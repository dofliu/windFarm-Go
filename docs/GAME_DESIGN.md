# 離岸風場・運維傳說 — 遊戲設計文件（GAME DESIGN, 權威版）

> 本文件是**現行遊戲模式的單一真實來源**。早期的 [GDD.md](GDD.md) 為歷史概念（勇者鬥惡龍式 tile-RPG），已不採用。
> 實作以 `src/` 與本文件為準。Last updated: 2026-07-08。

## 1. 定位
離岸風場 **運維（O&M）管理模擬 + 教學遊戲**。玩家經營一支運維團隊：調度船隊與技師、管理備品供應、在天氣窗內登塔排除故障，維持風場**可用率與發電量**。診斷與 SOP 為教學核心。風格參考《大航海時代》（船隊/港口/儀表）。

## 2. 核心循環（出勤就緒閘門 Mobilization Gate）
**關鍵原則：維修畫面不可直接進入。** 必須完成「出勤就緒檢查」並實際航行抵達機組後才能維修。

```
辦公室(母港大廳)
  → 接工單(調度中心) / 主線
  → 出勤就緒檢查（缺一不可）：
        ☐ 船舶 已指派且可用（船隊）
        ☐ 技師 對應科別且等級足夠、人數在船舶載量內（技師公會）
        ☐ 工具 達需求等級（機具工坊）
        ☐ 備品 在庫（否則下單 → 前置期到貨前無法維修）
        ☐ 天氣窗 可出航（海象 ≤ 船舶耐受度）
  → 出航：航行耗時(距離/船速) → 畫面顯示 ETA / 經過時數
  → 登塔：海象過大可能「登船延誤」→ 等待/改期/中止
  → 維修作業：診斷測驗 + SOP，在現場作業窗內完成
  → 返航 → 結算：可用率↑、發電量恢復、扣成本、KPI 更新、圖鑑解鎖
  → 重複；故障未修期間每日損失發電（停機成本）
```
**閘門 UI**：在辦公室時「維修作業」分頁**鎖定/停用**，顯示「尚未出勤抵達機組」；只有在「已抵達」狀態才解鎖。

## 3. 時間與天氣模型
- **日曆制**：以「天」為主時鐘；部分動作推進日期（出航、等待、休整、例行任務）。
- **航行時間**：距離 ÷ 船速 → 數小時（同日）或跨日（遠場/慢船）。
- **天氣窗 Weather Window**：每日海象隨機（可作業/警戒/停航）。
  - 出航需海象 ≤ 船舶耐受度；停航或超耐受 → 不能出發。
  - 抵達後「登船/登塔」需更嚴的浪高門檻 → 可能**登船延誤**（等下一個窗，日期推進）。
  - 現場維修有**作業窗倒數**（步驟/答錯消耗），窗關閉須撤離。
- **多日工期**：大修 = 航行 + 多次出勤 + 等天氣窗 + 備品前置期疊加 → 數小時～數週。
- **停機成本**：故障未修，每日損失發電與租船費。

## 5. 出勤就緒檢查（Mobilization Checklist）
出航前逐項檢核，缺項以紅字提示並阻擋出發：
1. **船舶**：已指派一艘狀態正常的船；其耐海象 ≥ 當前海象。
2. **技師**：工單故障科別 = 已指派技師科別；技師等級 ≥ 故障難度；在船人數 ≤ 船舶載量。
3. **工具**：工坊等級 ≥ 故障需求（或具備指定工具）。
4. **備品**：必備備品在庫；否則顯示「需下單，預計 N 天到貨」。
5. **天氣窗**：當日海象允許出航。
> 全部 ✓ → 「出航」可按；任一 ✗ → 指引玩家去對應設施準備。

## 6. 故障 ↔ 科別 / 備品 / 工具
每個故障定義：`discipline`(科別)、`part`(必備備品)、`toolReq`(工具需求)、`difficulty`(難度→技師等級門檻)。
範例：齒輪箱過熱 = 機械 / 齒輪箱油 / 一般工具 / Lv1；變流器過溫 = 電氣 / 變流器模組 / Lv2。

## 7. 績效 KPI 與排行
- **可用率 Availability (%)**：主指標；維修成功↑、停機↓。
- **發電量 Generation (MWh)**：機組運轉時累積，正比於可用率與裝置容量；停機流失。
- **營運成本 OPEX**：技師日薪、租船、備品、停機損失。
- **安全 Safety**：惡劣海象硬上、跳過步驟 → 安全事件扣分。
- **綜合績效分 Score = f(發電量, 可用率, −成本, 安全)** → 排行榜（之後接雲端供班級比較）。

## 8. 經濟
- **費用量級（◎ ≈ NT$，真實相對比例）**：耗材數萬（齒輪油/液壓油/碳刷/螺栓）；中型組件數十萬（偏航電機/海纜接頭）；大型組件百萬～近千萬（變槳軸承 180 萬、變流器 260 萬、葉片 900 萬）。
- **收入**：**每日售電收入（發電量 × 單價，主要現金流＝真正利潤來源，`dailyRevenue`）**、完成工單報酬（見下）、戰情室維運報酬、例行小任務、（選）備品交易價差。
- **支出**：備品採購＋倉儲維持費/折舊、招募費＋技師薪資、升級/保養（皆耗時，Phase B）、停機損失（見下）、船舶待命費、SLA 違約金。
- **人力（招募費 + 月薪，避免重複計勞動成本）**：技師可**招募**（一次性招募/GWO 訓練費 `200,000 + 等級×200,000`，耗時 1 天）與**解僱**（`FIRE`；出勤中不可）；勞動成本以**每日薪資** `salaryOf = 2,200 + 等級×700`（月薪約 **9~13 萬**）表示，**按在職天數自動分攤**——只僱半年就只付半年薪、解僱即停。人力規模成為「妥善率 vs 薪資」取捨（過度招募吃掉利潤，故解僱有意義）。
- **停機損失 → 現金（無雙重計算）**：戰情室機組停機（故障/維修中）每日損失其發電佔比，**直接折抵當日淨發電**＝同時減少**售電現金**與**發電量 KPI**（`advance` 內 `budget −= lost×price`、`generationMWh −= lost`）。框架：頭條可用率＝**合約上限**，戰情室停機把實際產出壓到上限之下；同一筆損失只計一次（不再另外從績效分扣 `fleetLostMWh`）。因此**放任機組故障會直接少賺錢**（模擬：放置策略由 +1122 萬翻轉為約 −20 萬，積極管理 +780 萬）。戰役工單期間的 `DOWNTIME_PER_DAY` 則仍是該關停機的代理（頭條可用率不隨單關變動）。
- **工單報酬＝備品核銷＋小額里程碑（避免雙重收入）**：完成維修的現金報酬設計為**約略涵蓋所消耗備品**（保固/保險核銷的概念）＋小額固定獎勵；**真正利潤來自修好後恢復的售電收入**。如此「修復獎勵」不會與「恢復發電的收入」重複計算，又保留遊戲必要的完成回饋（另計 XP）。
- **戰情室派船受海象限制**：海象超出船舶耐受度時無法派工/定檢（CTV 到警戒、SOV 到停航）；可改用遠端重啟或等可作業天氣窗（接上微觀天氣預報）。
- **戰情室派工消耗備品**：現場維修需消耗對應備品（`incidents.part` → `data.ts` 真實價格）；軟性故障可**免備品遠端重啟**。事件池加權（`incidents.weight`：耗材級常見、大型組件罕見），使「是否值得花 180 萬換軸承、或接受停機損失」成為真實 O&M 取捨。缺料時派工被擋（需先到交易所採購）。
- **出海航次成本（鼓勵批次維修）**：每趟出海收固定**動員/油耗費 `SORTIE_COST`（25 萬）＋船舶磨耗**，但**只在「目前沒有船在海上」時收**——同一航次再派多台維修免再收。因此「一部一部修」＝每次都付出海成本，**一趟同時修多台才划算**。這正是「多部機組故障時，綜合技師/工具/備品/船隻/天氣規劃最佳出海策略」的任務（就在戰情室，無需另設關卡）。
- **故障頻率**：`FAULT_RATE_BASE` 0.16→**0.09**、健康度加成 0.25→0.15（仍隨運轉比例縮放）——降低「修不勝修」感；放置 30 天妥善率約 7 成、積極管理可穩定接近 100%。挑戰改由經濟最佳化（航次/備品/薪資取捨）而非疲於奔命。
- 目的：累積預算 → 招募技師 / 升級船隊工具 / 備齊備品 → 承接更難工單 → 推進主線。

## 13. 登入 + 雲端紀錄 + 真排行榜
**目標**：學生登入後，系統記錄每位使用者的營運狀況（可用率/發電量/績效分/進度），匯整成**全班雲端排行榜**；教師可檢視。

> **最終採用方案：Google Apps Script Web App + 試算表（完全免費、免後端、免信用卡）。** 先試 Google 表單失敗（受管理帳號強制登入），改用 Apps Script Web App：`doPost` 寫入、`doGet` 回 JSON；含合理性驗證/簽章/節流防作弊（#35）。已實作：登入閘門(暱稱+班級碼)、結算自動送分、排行榜讀雲端、選配完整存檔同步(#31)。教師設定步驟見 **[LEADERBOARD_SETUP.md](LEADERBOARD_SETUP.md)**。下方 Supabase/Firebase 比較保留為歷史參考。

### 方案比較
| 方案 | 優點 | 缺點 | 建議 |
|---|---|---|---|
| **Supabase**（Postgres + Auth + Row Level Security）| 免自架伺服器、免費額度、JS SDK 直連、適合 GitHub Pages 靜態前端 | 需建專案、學 RLS | ⭐ 推薦 |
| Firebase（Firestore + Auth）| 同上、生態成熟 | NoSQL 查排行較繞 | 可 |
| 自建 FastAPI + DB | 完全掌控、貼合 NCUT 既有技術 | 要部署/維運伺服器 | 想自主時 |

→ **推薦 Supabase**：靜態前端最省力、最快上線。

### 登入方式（由輕到重）
1. **班級碼 + 暱稱**（最輕）：教師發一組班級碼，學生輸入暱稱即建立身分。無密碼、零摩擦，適合課堂。
2. **Google 登入**：實名、跨裝置、防重複；稍多設定。
3. Email magic-link：折衷。
→ 課堂首選 **①班級碼+暱稱**；要正式追蹤再加 ②。

### 資料模型（Supabase）
```
profiles(id, nickname, class_code, created_at)
scores(user_id, score, availability, generation_mwh, missions_done, day, updated_at)  -- 每人一列, upsert
-- 排行榜 = select * from scores order by score desc limit 50（可依 class_code 過濾）
```

### 整合點（前端）
- 啟動：若未登入 → 顯示登入畫面（班級碼+暱稱）。
- 存讀：登入後 GameData 與雲端同步（localStorage 為離線快取）。
- 結算事件（完成工單/推進日）→ upsert `scores`。
- 排行榜 modal：改讀雲端 `scores`（目前為本機 KPI）。

### 分期
- **P1（最小可用）**：班級碼+暱稱登入 → 關鍵事件 upsert 績效分 → 排行榜讀雲端。
- **P2**：完整存檔同步、Google 登入、教師後台（依 class_code 看全班）。

### 前置需求（需教師提供）
建立 Supabase 專案、取得 `SUPABASE_URL` 與 `anon key`（放前端環境變數）、建表與 RLS 政策。**取得金鑰後即可實作 P1。**

## 14. 進階擬真路線圖（規劃中）
朝「擬真運維模擬器」深化的方向（不需更動核心架構，多為在現有 `game.ts`/`tasks.ts`/`events.ts` 上擴充）：

1. **多回合工期（大修）**：✅ 已實作。重大故障（`MAJOR_FAULTS`，如發電機/主軸承）在現場拆檢（測驗+SOP）後轉入多回合大修 `Overhaul`，需累積 `OVERHAUL_NEED`（3）個「可作業天氣窗」工日；於母港逐日 `ADVANCE_OVERHAUL` 推進——可作業日 +1 工日，惡劣海象則停滯並付船舶待命費 `DEMURRAGE_PER_DAY`（demurrage）。完成後發放工單獎勵並大幅回復可用率。與 #2 預報結合：看準三日預報的可作業窗推進，避免在風暴中空燒待命費。見 `game.ts`、Hub「待處理工單」面板。
2. **微觀天氣預報**：✅ 已實作。天氣改為時間軸屬性（馬可夫慣性，風暴成群），每日「昨日預報」實現為今日海象（含 ~18% 預報誤差，教學：預報非百分百準確），並滾動產生未來三日預報。母港「風場動態」、出航就緒檢查、頂部 HUD 皆顯示三日預報＋風暴警示，支援預防性排程（預知風暴 → 提前搶修或靠港等更好的天氣窗）。見 `advanceWeather`/`makeForecast`/`forecast`、`Forecast.tsx`。
3. **合約 SLA**：✅ 已實作。設季度（`QUARTER_DAYS`=90 天）可用率底線 `SLA_FLOOR`（90%）；每日累計可用率，跨季結算平均，低於底線 → 扣違約金 `SLA_PENALTY` 並計入績效扣分（`computeScore` −25/次）。Hub「風場動態」顯示本季平均/底線/剩餘天數/風險，季末跳結算通知。逼出「戰略性放棄單機保全體」決策（與 #4 大修的停機取捨直接衝突，形成張力）。見 `advance()` 之 SLA 區段、`lastSla`。
4. **倉儲成本 / 備品折舊**：✅ 已實作。持有備品每日課徵倉儲維持費 `STORAGE_COST_PER_UNIT`（隨持有量上升），且每類備品每日有 `SPOIL_CHANCE` 機率折舊報廢 1 件（跳通知）。考驗 JIT vs 安全庫存：備愈多停機風險愈低，但維持費與耗損愈高。見 `advance()`、`dailyStorageCost`、Hub「備品庫存」面板。
5. **機組健康度（Health Index）**：✅ 已實作。全場健康度 `fleetHealth`（0-100）每日衰退（有未修故障時加速＝連鎖反應的根源）；越低 → `rollEvent` 觸發機率越高且偏向壞事件（連鎖故障）；維修(+8)/休整(+1.5)/良好預防決策可回復。見 `game.ts`/`events.ts`。（後續可加：每機獨立健康度、付費進階檢測）
6. **動態 SCADA 趨勢圖**：✅ 已實作。自由營運中心的輔助圖改為**即時動畫饋送**（趨勢/頻譜/雷達/長條隨時間變化）；付費解鎖 `DIAG_COST` 的**進階檢測** `diagLevel` 後，趨勢圖加上投影虛線、告警門檻線與風險讀數（如「約 N 步後達門檻」、BPFI 標註、胞 ETA），判讀更清晰。見 `OpsCenterModal.tsx` 的 `TaskChart`。
7. **技師疲勞 / 船舶保養週期**：✅ 已實作。技師加入 `fatigue`（0-100）：出勤/大修工日累積（`FATIGUE_PER_JOB`），每日休整回復（`FATIGUE_RECOVERY`）；達輪班上限 `FATIGUE_LIMIT` 不得派工（出航就緒檢查擋下，需休整或招募同科別技師輪班）。船舶加入 `vesselWear`：每趟出航累積，過高縮短維修作業窗，於 CTV 整備廠付費保養歸零。見 `availableEngineer`/`deployFatigue`/`vesselWindowPenalty`、Hub/技師公會/CTV 整備廠 UI。

## 15. 真實營運深化路線圖（學員回饋）
針對「更貼近真實風場營運」的回饋，分三期推進：

- **Phase A（快贏，✅ 已實作）**
  1. **出海動態航線俯瞰圖**：航行（enroute）時以港口→風場的俯瞰路線圖呈現船隻沿航線移動（取代純進度條），含航點、船跡、ETA。見 `RouteMap.tsx`、`SailScreen`。
  2. **每日遠端巡檢**：母港「遠端巡檢」動作 `REMOTE_CHECK` 消耗 1 天、累積經驗、早期偵測微幅回復健康度，作為計分週任務之間的日常節奏（其他小任務：調度中心例行、自由營運中心皆同樣消耗時間並累積績效）。

- **Phase B（時間與隨機，✅ 已實作）**
  - **統一時間成本**：除了既有會推進日的動作（休整/例行/沙盒/大修/遠檢），現在**維修出勤（FINISH_REPAIR/FAIL_REPAIR）= 1 天**、**招募訓練/設施升級/船舶保養/進階檢測 = 1 天**、**購置 SOV/拓展風場 = 2 天**；採購備品仍以到貨 lead time 表現時間成本（須在等待期間做其他事推進日）。發電量→排名既有（`generationMWh`/`computeScore`）。時間軸因此「總是往前」。
  - **每帳號機組隨機化（有界）**：計分主線保留同一週的故障種類與懸疑主線（公平＋劇情），但「哪一台機組故障」依帳號決定性隨機（`missionInstance`/`accountSeed`，以 baseUnit 為 key → m1/m6 對同一台機組的回扣一致）。各學員面對的基主不同、難度相近。見 `campaign.ts`。
  - 註：**故障種類的隨機**留待 Phase C 的機組個體事件層（並行隨機事件），以免破壞主線懸疑的共同根因伏筆。

- **Phase C（活體戰情層，✅ 核心已實作 / 採「新增層」而非重寫核心）**
  - 架構決策：**保留**現有戰役故事＋出海維修小遊戲＋自由營運中心不動，**新增**一個「風場戰情室」即時營運層（`FleetOpsModal`），#3b/#4/#5 都在這層落實，對既有平衡零破壞。
  - **機組個體模型**：每座風場 24 台機組（`Turbine`，狀態 正常/故障/維修中 + 發電佔比），`buildFleet`/`fleet`。開局植入數台故障使妥善率≈86%。
  - **並行工單 + 多組人力派遣**（回饋 #5）：`OPS_DISPATCH` 指派「對應科別且未過勞、未在工的技師」維修故障機組，多名技師 → 多筆 `opsJobs` 同時進行；技師有限 → 須排優先序。完工累積疲勞、計入 `fleetResolved`（績效 +8/台）。
  - **有界隨機事件**（回饋 #3b）：每日依健康度機率在隨機正常機組觸發隨機故障（`incidents.ts` 故障池），各局/各帳號面對的故障組合不同但難度相近。
  - **差異化停機損失**（回饋 #4）：故障/維修中機組每日損失其發電佔比，累積 `fleetLostMWh`（學習指標）——回應越慢、停機越久、損失越多。
  - 單一時間軸：戰情室「推進一天」`OPS_ADVANCE` 與其他所有耗時動作共用全域 day（天氣/SLA/疲勞/倉儲一起推進）。
  - **試玩後微調（已併入）**：
    - 故障頻率調降（`FAULT_RATE_BASE` 0.45→0.16、健康度加成 0.4→0.25、上限 0.6），不再修不勝修。
    - 新增**遠端重啟** `OPS_RESET`：軟性故障（偏航/變流器/感測器誤報，`incidents.resettable`）可遠端重啟清除——**免技師、1 天**（`REMOTE_RESET_DAYS`），讓有限人力專注於硬故障。
    - **經濟收入**：每日售電收入 = 發電量 × `ELECTRICITY_PRICE`（3000 ◎/MWh）併入現金流（`dailyRevenue`，Hub KPI 顯示「售電收入/日」）；戰情室每修復一台機組另給維運報酬 `FLEET_FIX_REWARD`（12 萬）。解決「現金花完後無收入」。
    - **故障平衡（模擬測試後）**：新故障機率改為隨「運轉中機組比例」縮放（只有運轉中的機組會新故障）→ 形成穩定平衡，避免死亡螺旋。模擬顯示：純放置 30 天妥善率仍 ≥40%（不再瞬間雪崩），單一技師約 60–70%，招齊各科別技師可達 ~100%（loop 可贏且獎勵投資人力）。
  - **C2（已實作）**：
    - **船舶並行上限** `vesselJobCap`（CTV 2、SOV 4、每整備等級 +1；遠端重啟不占用）——現場工單同時數受船舶限制，給船隊升級/購置 SOV 在戰情室一個明確用途。
    - **預防性定檢** `OPS_INSPECT`：派一組人巡檢（`INSPECT_DAYS` 天）→ 完成後 `INSPECT_BUFF_DAYS` 天內故障率 ×`INSPECT_FAULT_MULT`（降低）＋小幅健康度回復，教「預防保養減少故障」。
  - **經濟整合（已實作）**：戰情室停機**直接折抵淨發電** → 同時減少售電現金與發電量 KPI（見 §8）；績效分用淨發電量（不再額外扣 `fleetLostMWh`，避免重複）。「忽略機組故障」因此**同時少賺錢、又掉排名**（模擬：放置 ≈ −20 萬／績效 5966；積極管理 +780 萬／績效 10500）。`fleetLostMWh` 保留為「累積停機損失」學習指標。
  - **平衡模擬器**：`npm run sim`（`test/sim.mjs`，可帶天數/種子）以 passive/active/full-crew 三策略輸出妥善率/經濟/績效，調數值後可快速回測手感。
  - **C2 後續**：✅ 已把隨機事件池綁到「每週主題」（見 §16 每週主題挑戰）。

## 17. 出海決策支援・學習迴路收斂・無障礙（本輪新增）
延續「作業安全窗（work window）」機制，補上「出海前判斷」與「出海後止損」的完整決策鏈，並收斂學習迴路（檢索練習/復盤）與無障礙基礎。

- **出海前工期預估 vs 天氣窗**（`SailScreen`）：出勤就緒檢查新增「⏱ 工期預估 vs 天氣窗」卡 —— 列出航線(單程/往返)、登船、檢查/診斷、維修 SOP 的預估耗時，對比 `workWindowMax()`(與現場維修**共用同一公式**，兩處數字永遠一致)算出的天氣窗允許時段，並算出「保留餘裕」(充足/偏緊/不足)。同卡另附**判斷提醒**：船舶磨耗縮窗、技師疲勞接近輪班上限、今日不利但預報有可作業日時建議「靠港等窗」。只提示、不擋出航 —— 由玩家自行判斷。
- **半途成果保留（#carry，`REPLAN_RETURN`）**：維修中途作業窗吃緊時，除「繼續作業」外新增**「回港再規劃」**：進 1 天、**不計安全事件**、已完成的診斷/SOP 步驟**保留**(工作交接紀錄) —— 擇日重新登塔只需重擲作業窗續修。與「作業窗關閉才撤離」(`FAIL_REPAIR`：進度全失 + 計 1 次安全事件)明確區隔，獎勵「看準天氣窗、及時止損」優於「硬撐到窗關」。母港工單、登船畫面、出海工期預估(只算剩餘工作)全鏈路呈現保留狀態。
- **加班搶修（#rush，`RUSH_SOP`）**：維修不利時的第三個選項 —— 剩餘 SOP **一次趕完、總耗時減半**(`rushCost`)，但有 `RUSH_RISK`(25%)機率觸發安全近失事件(計入 `safetyIncidents`)。僅在「維修不利」時出現(緊急槓桿、非常規最佳解)，與「繼續作業 / 回港再規劃」構成三選一：**搶修(快、險) vs 續作(穩、可能窗關) vs 返港(慢、保進度)**。風險由 UI 擲骰、reducer 保持確定性(可測、fuzz 友善)。
- **任務復盤（#debrief）**：完工時經理給「出海復盤」：星級(依剩餘作業窗餘裕；診斷答錯 ≥2 次扣星) + 量化摘要(窗 N 時段、用 X、餘 Y) + 一句 takeaway，把每次出海變成可檢討的學習點。
- **診斷連對 streak（#streak）**：`GameData.answerStreak` 於 `RECORD_ANSWER`(僅計首次作答)連對 +1、答錯歸零；每次答對加 XP = `streak×2`(封頂 10)。維修診斷面板與自由營運中心皆顯示 🔥「連對 N」徽章與里程碑通知(3 題/每 5 題)，鼓勵「出手前先思考」而非亂猜。
- **錯題本主動回想（`MistakeLog`）**：複習前先「自我測驗」—— 兩選項(正解 vs 你當時的錯誤選擇)隱藏正解，答對/答錯後才揭曉正解與教訓、再寫維修檢討。檢索練習(active recall)比只看反思更利長期記憶保留。
- **無障礙**：`useReducedMotion()` 尊重 `prefers-reduced-motion` 系統偏好，並支援個人檔案頁「🎬 減少動態」手動開關(localStorage 覆寫)——場景影片改靜態首幀、CSS 動畫停用。海象/故障狀態改用**圖示 + 顏色雙重編碼**(☀/⛅/🌀、⚠ 故障徽章)，不只靠顏色區分(色覺友善)。

## 18. 人力短缺實效化・軸承備品正名（本輪新增）
接續 §17 的內容查核，補上兩處**邏輯完整性**缺口——把「只影響展示、不影響決策」的事件接上實際機制，並修正一組權宜共用的備品命名。

- **人力短缺/罷工事件實效化（#crew，`crewShortfallJobs`/`effectiveJobCapOf`）**：`crew_shortage`(−3)、`strike`(−6) 事件原本只調整展示用 `techAvail`，`OPS_DISPATCH`/`OPS_INSPECT` 完全不讀 → 事件承諾了後果卻不兌現。現在缺工**直接折抵「可同時開的現場作業面」**：`crewShortfallJobs = ⌈(techTotal − techAvail) / CREW_PER_JOB⌉`（`CREW_PER_JOB=4`），有效上限 `effectiveJobCapOf = max(1, jobCapOf − crewShortfallJobs)`。設計要點：
  - **滿編不懲罰**：以「缺額」而非「絕對人數」計，`techAvail=techTotal` 時 shortfall=0 → 不影響任何船隊（含頂級母船）；缺工愈多、可並行的作業面愈少。
  - **下限 1**：極端缺工仍可派一組(短手應變)，且**遠端重啟不占名額、主線工單不受此限** → 不會硬鎖遊戲；`techAvail` 每日休整 +1 逐步回復，讓「罷工」有明確的止血與恢復節奏。
  - **開局滿編 30/30**（原 24/30）＋ tech 升級同步 `techAvail`(+2)：避免「憑空缺額」在開局或擴編後誤觸折抵。
  - **戰情室透明化**（`FleetOpsModal`）：新增「可出勤班組」統計；短手時顯示 👷 橫幅（`vesselCap → cap`、少 N 面）與「現場已滿」時的成因區分（人力 vs 船舶）。事件不再是純展示，而是「先遠端重啟軟故障、批次搶修、靠港補人」的真實取捨。
- **軸承備品正名（#bearing，`drive_bearing`）**：主軸承振動事件(`bearing`)與發電機振動圖鑑故障(`gen_vibration`)原本權宜共用「變槳軸承」(`pitch_bearing`)備品（規避 Tier 閘門）。新增 Tier 3 專屬備品**「傳動軸承組(發電機/主軸)」**(`drive_bearing`, ◎130 萬)並重新指派兩者——與 Tier 4「主軸承」全換(`main_bearing`)語意區隔（振動維修級 vs 重吊更換）。同時新增 `pitch_bearing_wear`（變槳軸承磨耗）事件，讓 `pitch_bearing` 備品保有真實消費端（與既有 `cs_pitch_bearing_wear` 案例研究呼應），不成孤兒。
