# 壓力測試報告 — 多人併發提交 / Concurrent-Load Stress Test

> 目的:回答「多位學生同時提交成績/存檔請求時,雲端系統是否撐得住?」
> 執行:`npm run stress`(預設 200 人 / 10 秒 / 伺服器同時執行上限 30);可帶參數 `node test/stress.mjs [users] [seconds] [serverConcurrency]`。
> Last run: 2026-06-25.

## 為什麼用「忠實 mock」而非直接打線上端點
正式後端是 **Google Apps Script Web App**(免費、免自架、見 `docs/leaderboard-appsscript/Code.gs`)。直接對 Google 的 `/exec` 灌併發流量會踩到 Google 配額、且不應對線上服務壓測。因此 `test/stress.mjs` 在本機**忠實重現後端契約**:
- **djb2 簽章驗證**(與 `src/cloud/sheet.ts` 同一密鑰/格式)
- **clampInt 合理性夾值**(擋異常分數,與前後端一致)
- **8 秒節流去重**(對應 Code.gs `CacheService` 視窗,防作弊/省寫入)
- **排行榜彙整**(每班/暱稱取最高分、排序、取前 50)
- **Apps Script 同時執行上限**:以 semaphore 模擬(超過上限的請求**排隊**而非丟棄,反映真實行為)

客戶端送分邏輯比照 `src/cloud/sheet.ts` 的 `sanitize` + `sign`。

## 結果(代表性數據)

| 情境 | 吞吐量 | p50 / p95 / p99 延遲 | 同時執行 | 最大排隊 | 簽章錯誤 | 其他錯誤 | 判定 |
|---|---|---|---|---|---|---|---|
| 200 人 / 10s / cap 30 | ~705 req/s | 283 / 315 / 352 ms | 30(=上限) | 170 | 0 | 0 | ✓ |
| 500 人 / 10s / cap 30 | — | 718 / 772 / 943 ms | 30(=上限) | 470 | 0 | 0 | ✓ |

**關鍵觀察**
1. **8 秒節流大幅削減寫入**:200 人持續猛送時,約 **74% 的送分被去重**(throttled),只有最新一筆真正寫入試算表。這正是保護後端的設計——真實情況學生只在「結算/推進日」事件送分(頻率遠低於壓測),寫入負載會更低。
2. **併發有界、不丟請求**:同時執行數恆守在上限(30),超出的請求**排隊**處理,無丟棄、無 5xx;簽章與夾值在併發下仍正確(bad-sig = 0)。
3. **延遲隨飽和優雅升高**:人數增加 → 排隊變深 → 延遲上升(p99 由 ~0.35s 升到 ~0.94s),但系統穩定不崩。對「送分/讀排行」這類非互動關鍵路徑,秒級延遲完全可接受。

## 真實後端的容量特性與結論
- **客戶端為射後不理(no-cors fire-and-forget)+ 離線快取**:送分/存檔失敗或延遲都不影響遊戲進行(`src/cloud/sheet.ts`、`src/cloud/api.ts`)。即使後端慢或暫時不可用,玩家無感,稍後自動補送。
- **雲端為主 + 本機快取(較新者為準)**:跨裝置一致、離線可玩(`pickNewer`)。
- **Apps Script 配額**:免費方案有每日請求/執行時間額度與同時執行上限。**8 秒節流**把「每人每 8 秒最多 1 筆寫入」當作天然限流,單班 30–50 人、多班並行的教學場景在配額內綽綽有餘。

**結論:在教學場景(單班數十人、偶發送分)下系統可輕鬆負荷;即使刻意以數百人猛灌,契約仍守住(併發有界、不丟請求、簽章/節流正確),延遲僅優雅升高。**

## 建議(未來如需更大規模)
- 若要支援「全校同時、且每秒多筆」的極端場景,可改用具水平擴展的後端(Supabase/Firebase,見 GAME_DESIGN §13 比較),把寫入批次化或加佇列。
- 維持客戶端 no-cors + 離線快取設計(目前已是),這是抵抗後端抖動的關鍵。
- 可考慮把送分頻率進一步綁定「關鍵結算事件」而非定時,降低尖峰寫入。
