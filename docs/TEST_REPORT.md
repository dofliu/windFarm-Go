- **結果**:✅ 零型別錯誤。

- **涵蓋範圍**(依模組分區):

  | 戰情室/派工 | 科別配對派工、遠端重啟、船舶並行上限、出海動員費、定檢 buff | 派工規則、批次出海不重複收費、備品消耗 |

## 4. 平衡模擬 — `npm run sim`(120 天)

- **項目**:驗證「投入越多→成效越好」的進程激勵設計成立,且長線不崩盤。
- **方法**:`test/sim.mjs` 以三種玩法策略各推進 120 遊戲日,量測妥善率、停機損失與績效分。
- **結果**(代表性):

  | 策略 | 妥善率(現/均/低) | 停機損失 MWh | 績效分 |
  |---|---|---|---|
  | passive(放任) | 58 / 87 / 54 % | 1965 | 12145 |
  | active(管理) | 79 / 94 / 71 % | 875 | 13771 |
  | full-crew(全配技師) | 100 / 97 / 83 % | 465 | **14961** |

  → 三策略績效**單調遞增**(放任 < 管理 < 全配),且放任策略仍不崩盤(均妥善率 87%),符合「入門友善、精進有回報」的設計。

## 5. 併發壓力測試 — `npm run stress`

- **項目**:模擬「多位學生同時提交成績/存檔、讀取排行榜」,檢驗雲端後端契約在併發下是否守住,以及併發鎖是否消除競態。
- **方法**:`test/stress.mjs` 在本機**忠實重現** Google Apps Script 後端契約(djb2 簽章驗證、`clampInt` 夾值、8 秒節流去重、排行榜彙整、同時執行上限以 semaphore 模擬「排隊而非丟棄」),再以併發負載打它。**不直接壓測線上 `/exec`**(避免踩 Google 配額、不對線上服務灌流量)。詳見 [`STRESS_TEST.md`](STRESS_TEST.md)。
- **結果**:

  | 情境 | 吞吐量 | 延遲 p50/p95/p99 | 簽章錯 | 其他錯 | 同時執行 | 最大排隊 | 判定 |
  |---|---|---|---|---|---|---|---|
  | 50 人 / 10s / cap 30(單班真實) | 706 req/s | 68 / 97 / 104 ms | 0 | 0 | 30(=上限) | 20 | ✅ |
  | 200 人 / 10s / cap 30(預設) | 701 req/s | 284 / 315 / 348 ms | 0 | 0 | 30 | 170 | ✅ |
  | 500 人 / 12s / cap 30(超載) | 747 req/s | 712 / 759 / 913 ms | 0 | 0 | 30 | 470 | ✅ |

  **關鍵觀察**
  1. 超過後端同時執行上限的請求**排隊而非丟棄**(無 5xx、無遺失)。
  2. 簽章驗證、`clampInt` 夾值、8 秒去重(防作弊/省寫入)在併發下全部正確;`bad-sig = 0`。
  3. 延遲隨飽和**優雅升高**(p99 由 ~0.1s → ~0.9s),系統穩定不崩。對「送分/讀排行」這類非互動關鍵路徑,秒級延遲可接受。

  **併發競態示範**(對應 `Code.gs` v2.1 `LockService`):200 個併發「讀-改-寫」同一筆紀錄

  | 條件 | 最終計數(應為 200) |
  |---|---|
  | 無鎖 | **1** ← lost updates,多人互相覆蓋 |
  | 有 `LockService`(Code.gs v2.1) | **200** ← 序列化、無遺失 |

  → 確認雲端併發鎖有效消除了無鎖路徑會發生的更新遺失。

## 6. 線上後端實打 — `npm run live-check`

- **項目**:對**實際部署**的 Apps Script 後端做端到端檢查(註冊/登入/存檔/送分/讀排行 + 併發鎖證明,共 10 項)。
- **方法**:`test/live-check.mjs` 直接呼叫線上 Web App;預設唯讀,加 `--write` 才寫測試資料。
